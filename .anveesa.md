# anveesa-cli project instructions

You are working inside the **anveesa-cli** repository — the source code for the anveesa CLI tool itself.

## What this project is

A Rust CLI (edition 2024) that provides a multi-provider terminal AI assistant. It compiles to a single binary called `anveesa`. Version 0.7.0. 181 tests. Zero warnings.

## Language and toolchain

- **Rust** (edition 2024, MSRV: stable)
- Build: `cargo build --release`
- Test: `cargo test`
- Lint: `cargo clippy -- -D warnings` (must be clean on Linux/Ubuntu)
- Format: `cargo fmt` (enforced by CI)

## Module structure

| Module | Purpose |
|---|---|
| `src/lib.rs` | CLI dispatch, interactive REPL, one-shot mode |
| `src/config.rs` | Configuration loading and types |
| `src/tools.rs` | 28 AI tool implementations |
| `src/provider/openai_compatible.rs` | Core LLM provider: SSE streaming, tool dispatch, retry |
| `src/tui.rs` + `src/tui/` | Full-screen TUI (ratatui) |
| `src/web.rs` | axum browser chat UI |
| `src/session.rs` | Conversation persistence |
| `src/workspace.rs` | Context injection (git, files, README) |
| `src/display.rs` | Plain-terminal output helpers |
| `src/image.rs` | Clipboard and file image handling |
| `src/prompt.rs` | Raw terminal line reader |

## TUI App sub-structs

When modifying TUI code, fields are grouped — use the right sub-struct:

- `app.kbd.*` — keyboard input state
- `app.live.*` — streaming / live turn state  
- `app.conv.*` — conversation history and session
- `app.view.*` — display messages, scroll, search

## Critical invariants

- **Never** drain `app.conv.history` without first inserting a `ChatMessage::user("[Context note: ...]")` breadcrumb — the model loses context otherwise.
- **Never** skip `cargo fmt` before committing — CI will fail.
- **Always** run `cargo clippy -- -D warnings` locally before pushing — Ubuntu clippy catches things macOS misses.

## Slash commands available in the TUI

`/add <file>`, `/diff`, `/commit [msg]`, `/memory <note>`, `/compact`, `/undo`, `/copy`, `/export`, `/model`, `/provider`, `/status`, `/search`, `/clear`, `/help`, `/exit`

## Release process

1. Bump `version` in `Cargo.toml` and `package.json`
2. `git tag vX.Y.Z && git push origin main --tags`
3. GitHub Actions builds binaries for 5 platforms and publishes to GitHub Release
4. `npm publish` from repo root (or automated via `NPM_TOKEN` GitHub secret)

## npm package

The npm package lives at the **root** `package.json` (not in `npm/`). The `scripts/install.js` postinstall downloads the platform binary from GitHub releases, falling back to `cargo build --release`.

## Do not

- Add dependencies to Cargo.toml without a clear reason
- Change the `workspace_context_for` function to read file contents (would blow up context window)
- Remove the `#[allow(unused_assignments)]` on `last_effective_model` without understanding why it's there
